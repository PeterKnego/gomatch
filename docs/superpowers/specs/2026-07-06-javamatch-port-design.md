# javamatch — Java port of gomatch

Date: 2026-07-06
Status: approved

## Goal

Port the `gomatch` project (a single-instrument limit order book matching
engine running as a Go ClusteredService on Aeron Cluster) to Java, with the
same structure and comparable performance, so the two implementations can be
benchmarked against each other.

Reference implementation: `/home/claude/ultima/gomatch` (Go, aeron-go fork
`PeterKnego/aeron-go` v0.2.0).

## Approach

**Structural mirror with idiomatic mechanics.** Packages, classes, methods,
and the event model mirror gomatch 1:1, but the mechanics use Java-native
Aeron/SBE the way they are meant to be used: sbe-tool flyweight codecs decoded
in place (no intermediate copy), reused encode buffers, Agrona
`Long2ObjectHashMap` for the order map. The result is "the same program,
written by a competent Java/Aeron developer" — the fair basis for a
performance comparison.

Rejected alternatives:
- *Literal transliteration* (per-message allocations, buffer copies copied
  from Go): would make Java look artificially slow; nobody writes Aeron Java
  that way.
- *Fully idiomatic zero-garbage* (callback-based engine, no Event objects):
  fastest possible but diverges structurally; apples-to-oranges comparison.

## Wire and state compatibility

- SBE schema id 901 (`gomatch-schema.xml`) is copied verbatim; only the
  generated-code namespace changes. The Go and Java engines are
  **wire-compatible**: either loadgen can benchmark either engine.
- The snapshot format is byte-identical (`GMS1` magic, version 1,
  little-endian, orders in bids-best-first-FIFO-then-asks order). A Java
  engine can restore a Go snapshot and vice versa.

## Project setup

- Location: `/home/claude/ultima/javamatch`, its own git repository.
- Build: Gradle (wrapper committed), Java 21 toolchain.
- Dependencies: `io.aeron:aeron-all:1.52.0` (same version gomatch systests
  use; bundles Agrona), JUnit 5 for tests, `uk.co.real-logic:sbe-all:1.38.1`
  as a codegen-time tool (same sbe version as gomatch's `generate.sh`).
- SBE codecs are generated by a Gradle task from
  `src/main/resources/gomatch-schema.xml` into a generated-sources dir,
  compiled with the main source set. Generated code is not committed.

## Components

### engine

Line-for-line port of the matching core: no Aeron imports, no I/O, integer
ticks only, fully deterministic.

- `OrderBook`: sorted `ArrayList<PriceLevel>` per side (bids descending, asks
  ascending), binary search (`levelIndex`) for level lookup/insertion,
  intrusive doubly-linked FIFO of orders within a level, Agrona
  `Long2ObjectHashMap<Order>` for the id→order map. `nextOrderId` sequence
  identical to Go.
- `Event`: flat tagged-union class mirroring Go's `Event` struct; engine
  methods return `List<Event>` (allocation per command mirrors Go's slice
  allocation; this is part of the structural contract).
- Event emission order, book-update dedup ("touched levels in order of first
  change"), and reject semantics identical to Go.
- Snapshot: `writeTo(...)`/`restoreFrom(...)` producing/consuming the exact
  Go byte format (little-endian, magic `0x474D5331`, version 1).

### service

`MatchingService implements io.aeron.cluster.service.ClusteredService`:

- `onSessionMessage`: read the 8-byte SBE header, verify schema id 901,
  dispatch template 1 (NewOrder) / 2 (CancelOrder) via reused flyweight
  decoders directly over the ingress buffer, drive the engine, route events.
- Routing identical to Go: ExecutionReports to the owning session;
  TradeEvent and BookUpdate broadcast to all sessions in insertion order
  (`sessions` map + insertion-ordered session id list).
- Egress encoding into a reused `ExpandableDirectByteBuffer` via reused
  flyweight encoders; `offer` retries with the cluster idle strategy on
  back-pressure/admin-action, logs and drops on other errors.
- Snapshot: `onTakeSnapshot` writes the engine snapshot in 1024-byte chunks;
  `onStart` drains the snapshot image and restores.

### client

`MatchClient` over `io.aeron.cluster.client.AeronCluster`:

- `connect(aeronDir, ingressEndpoints, listener)` and
  `connectWithEgress(..., egressEndpoint, ...)` mirroring Go, same channel
  setup (`aeron:udp?alias=...`, configurable egress endpoint), 30 s connect
  deadline.
- `submitOrder`, `cancelOrder` encode with reused flyweights into a reused
  buffer, offer with 10 s deadline and idle on back-pressure.
- Egress adapter decodes ExecutionReport/TradeEvent/BookUpdate into typed
  records delivered to a `Listener` interface (same three callbacks).

### app

- `EngineMain`: launches a `ClusteredServiceContainer` hosting
  `MatchingService`, honoring `AERON_DIR` and `CLUSTER_DIR` env vars like the
  Go `cmd/engine`.
- `LoadGen`: same flags (`-orders`, `-rate`, `-aeron-dir`, `-ingress`,
  `-egress`), same deterministic scenario (alternating sides, prices 98–102
  straddling mid, qty 1–10, seeded RNG), same coordinated-omission-corrected
  latency measurement from *scheduled* send time, same output format
  (throughput line + p50/p99/p99.9 ack latency).

## Testing

- Unit tests (JUnit 5) port every gomatch unit test: engine matching,
  order-book, cancel, snapshot round-trip + bad-magic, determinism; codec
  round-trips; service encode/dispatch tests; client egress decode test.
- `src/systest/java` source set, run by a dedicated `systest` Gradle task
  (excluded from plain `test`): launches `ClusteredMediaDriver` **in-process**
  (per-run port blocks derived from pid + instance counter, temp dirs) plus a
  `ClusteredServiceContainer`, then runs the same two scenarios as gomatch:
  match + market data, and resting-order-survives-snapshot-restart.

## Benchmark plan

1. Single-node cluster + Java engine on this machine; run `LoadGen` with the
   same parameters used for gomatch (e.g. 100k orders open-loop, and paced
   `-rate` runs for honest latency percentiles).
2. Same scenario with the Go engine for the side-by-side numbers.
3. Cross-run (Go loadgen → Java engine) as a wire-compatibility sanity check.

Comparability rules: same machine, same media driver configuration, same
order scenario, same measurement methodology (ack latency from scheduled send
time).

## Error handling

Same policies as Go: unknown schema id logged and ignored; unknown template
id ignored; decode errors logged and dropped; snapshot restore failure is
fatal at startup; egress offer failures other than back-pressure/admin-action
are logged and the frame dropped.

## Known limitations

Inherits gomatch's v1 limitation: if only the service container restarts
while the consensus module keeps running, sessions restored from a cluster
snapshot are not replayed through `onSessionOpen`; a full node restart does
not have this problem.
